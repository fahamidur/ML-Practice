Execution Engine

Fixed day modes

The learning system is stronger when the day itself has a clear job. The content changes by week, but the daily intent remains stable so execution does not depend on mood.

Monday · Learn and start building

110 to 150 minutes

Purpose

Reading Rule
Read the week’s main C++ and PyTorch resources in short focused blocks. Stop reading as soon as the first implementation can start.

Build Rule
Write the first runnable version immediately, even if it is rough.

Expected Output
One fresh file must run before the session ends.

Tuesday · Rebuild from memory

100 to 140 minutes

Purpose
Force recall so the learner sees what is actually understood.

Reading Rule
Use notes only after the first recall attempt fails.

Build Rule
Recreate one C++ file and one PyTorch file without copying.

Expected Output
A memory-built version with comments on what was forgotten.

Wednesday · Hard drill and debugging

100 to 150 minutes

Purpose
Take the week’s hardest drill and explain every breakage clearly.

Reading Rule
Read only enough to unblock the next mistake.

Build Rule
Push through one difficult drill and log the bug-fix chain.

Expected Output
One solved hard drill plus one debugging note.

Thursday · Mini build day

110 to 160 minutes

Purpose
Combine the week’s pieces into something slightly larger and cleaner.

Reading Rule
Use the build-system or project-structure resource if the week needs it.

Build Rule
Turn loose exercises into one mini project or module.

Expected Output
A working mini build with a sensible folder layout.

Friday · Refactor and verify

90 to 140 minutes

Purpose
Improve names, structure, tests, and reliability.

Reading Rule
Read the reference docs only for exact fixes or API details.

Build Rule
Refactor aggressively and add checks, tests, or assertions.

Expected Output
Cleaner code plus visible verification, tests, reload, or sanity checks.

Saturday · Package and push

75 to 120 minutes

Purpose
Turn the week into something that can be shown later.

Reading Rule
Review notes and README drafts only.

Build Rule
Organise files, write summary notes, and push to GitHub.

Expected Output
A clean commit, a short technical summary, and a clear next-week setup.

Sunday · Light review or rest

20 to 45 minutes

Purpose
Protect consistency without adding cognitive overload.

Reading Rule
Read notes, skim one old file, or rest completely.

Build Rule
No heavy building. Only micro-review if energy is low.

Expected Output
At most a note update or one tiny review drill.

Portfolio Projects

Projects that prove engineering practice

These projects exist to show that the learner can move from exercises into structured, reproducible, and explainable technical work.

Project 1 · PyTorch Training Toolkit

Portfolio

A reusable training repository that proves strong Python-side engineering discipline rather than notebook-only experimentation.
  • src, configs, models, data, scripts, tests, and README folders
  • custom dataset or loader logic
  • validation loop, checkpointing, and metrics logging
  • plotting or error-analysis output
  • one clean example command that another person can run

Proof of quality
The repository should run from a clean environment and produce a visible training artefact, metrics file, and saved weights.

Project 2 · C++ Utility Library for AI Work

Portfolio

A small but serious C++ library showing build discipline, testing, ownership clarity, and reusable tooling habits.
  • CSV reader or parser
  • timer and logger utilities
  • basic config reader
  • unit tests wired through CMake
  • clear build and test instructions

Proof of quality
A fresh clone should build cleanly, run tests from the terminal, and expose simple reusable components.

Project 3 · Python to C++ Bridge Project

Portfolio

The portfolio piece that demonstrates where model work meets systems-style implementation.
  • option A, tiny C++ inference application or option B, small custom operator
  • correctness comparison against a Python baseline
  • design note explaining the boundary between Python and C++
  • final README with exact build and run steps

Proof of quality
The project should make the Python-C++ seam visible and technically understandable, not merely present code.
